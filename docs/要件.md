# 予定調整アプリ 要件定義書

## 1. アプリの目的

複数人で予定を合わせるために、主催者が候補日時を作成し、参加者にURLを共有する。
参加者はログイン不要で、自分の名前を入力し、行ける候補日時を選んで回答する。

主催者は管理画面から、各候補日時ごとの参加可能人数や回答状況を確認し、最終的な日程決定に使う。

---

# 2. 基本方針

## 2.1 最優先する体験

このアプリで一番大事なのは、参加者が迷わず回答できること。

参加者側は、

1. URLを開く
2. 名前を入れる
3. 行ける日を押す
4. 送信する

だけで完了する。

アカウント登録、Googleログイン、カレンダー連携、メール認証などは不要。

---

## 2.2 ログインの扱い

ログインが必要なのは **管理者だけ**。

| 利用者 | ログイン | できること                 |
| --- | ---: | --------------------- |
| 管理者 |   必要 | イベント作成、編集、削除、回答確認     |
| 回答者 |   不要 | 名前入力、候補日時への回答、自分の回答修正 |

管理者ログインは、メールアドレスとパスワードでよいです。Supabase Auth はメール/パスワード認証に対応しており、Next.js のSSR構成でも使えるため、この用途には合っています。([Supabase][1])

---

# 3. MVPの範囲

まず作るべき最小機能はこれです。

## 3.1 管理者側

管理者は以下ができる。

1. ログインする
2. 予定調整イベントを作成する
3. タイトル、詳細、候補日時を設定する
4. 招待URLをコピーする
5. 回答一覧を見る
6. 各候補日時の参加可能人数を見る
7. イベントを締め切る
8. イベントを編集する
9. イベントを削除する

---

## 3.2 回答者側

回答者は以下ができる。

1. 招待URLを開く
2. イベントのタイトル、詳細、候補日時を見る
3. 名前を入力する
4. 行ける候補日時を選ぶ
5. 回答を送信する
6. 送信後、自分の回答を修正する

---

初期版では入れない方がよいもの。

| 機能            | 理由          |
| ------------- | ----------- |
| Googleカレンダー連携 | 実装が急に複雑になる  |
| 回答者ログイン       | 仲間内用途には重い   |
| メール通知         | MVPでは不要     |
| LINE通知        | 後回しでよい      |
| 出欠コメント欄       | 最初はなくても成立する |
| 投票期限リマインド     | 後回し         |
| 複数管理者         | 最初は1人でよい    |
| 決済            | 不要          |
| 公開イベント検索      | 不要          |

# 5. 主要ユースケース

## UC-001 管理者が予定調整イベントを作る

### 流れ

1. 管理者がログインする
2. 「新しい予定を作成」を押す
3. タイトルを入力する
4. 詳細を入力する
5. 候補日時を追加する
6. 「作成」を押す
7. 招待URLが発行される
8. 管理者がURLをコピーしてLINEなどで共有する

### 入力項目

| 項目        | 必須 | 内容             |
| --------- | -: | -------------- |
| タイトル      | 必須 | 例：5月飲み会の日程調整   |
| 詳細        | 任意 | 例：新宿あたりで考えています |
| 候補日時      | 必須 | 複数登録可能         |
| 締切日時      | 任意 | 回答期限           |
| 回答結果の公開設定 | 任意 | 回答者にも結果を見せるか   |
| 回答ステータス   | 任意 | ○/×のみ、または○/△/× |

### MVPでの推奨

最初は **○/×のみ** でよいです。
ただしDB上は後から `maybe` を追加できるようにしておくとよいです。

---

## UC-002 回答者が予定に回答する

### 流れ

1. 回答者が招待URLを開く
2. イベント内容を見る
3. 名前を入力する
4. 行ける候補日時をタップする
5. 「回答する」を押す
6. 完了画面が表示される
7. 自分の回答を後から修正できる

### 回答画面の理想

スマホで以下のように表示する。

```text
5月飲み会の日程調整

新宿あたりで考えています。
候補の中から行ける日を選んでください。

あなたの名前
[ 山田太郎            ]

候補日時
[ 5/10 金 19:00〜 ]  ○ 行ける
[ 5/11 土 18:00〜 ]  × 行けない
[ 5/17 金 19:30〜 ]  ○ 行ける

[ 回答する ]
```

ポイントは、表形式よりも **カード形式** の方がスマホで押しやすいことです。

---

## UC-003 回答者が自分の回答を修正する

ログインなしで修正できるようにするには、名前だけでは不十分です。
同じ名前の人がいたり、他人が勝手に編集できたりするからです。

そのため、回答送信時に **編集用トークン** を発行します。

### 推奨仕様

回答後に、ブラウザの `localStorage` に以下を保存する。

```text
event_id
respondent_id
edit_token
```

同じブラウザで再度URLを開いた場合、

```text
あなたはすでに回答済みです。
[ 回答を修正する ]
```

と表示する。

### さらに良い仕様

回答完了画面で、

```text
この回答を後から修正できます。
この端末であれば、同じURLから修正できます。

別の端末で修正したい場合はこちら：
[ 編集用リンクをコピー ]
```

を出す。

編集用リンクにはトークンを含める。
ただし、DBには生トークンを保存せず、ハッシュ化したトークンだけ保存する。

# 7. データ設計

Supabaseを使うなら、Postgresのテーブルとして以下を作るのがよいです。SupabaseはPostgresのRLSを使って行単位のアクセス制御ができ、Authと組み合わせて認可設計ができます。([Supabase][2])

## 7.1 events

予定調整イベント本体。

| カラム                  | 型           | 説明                               |
| -------------------- | ----------- | -------------------------------- |
| id                   | uuid        | 主キー                              |
| owner_user_id        | uuid        | 管理者ユーザーID                        |
| public_slug          | text        | 公開URL用のランダム文字列                   |
| title                | text        | タイトル                             |
| description          | text        | 詳細                               |
| admin_note           | text        | 管理者用メモ                           |
| timezone             | text        | 例：Asia/Tokyo                     |
| status               | text        | draft / open / closed / archived |
| visibility           | text        | public_result / private_result   |
| allow_maybe          | boolean     | △を許可するか                          |
| response_deadline_at | timestamptz | 回答期限                             |
| created_at           | timestamptz | 作成日時                             |
| updated_at           | timestamptz | 更新日時                             |
| deleted_at           | timestamptz | 論理削除                             |

### public_slug

`public_slug` は連番にしない。
推測されにくいランダム文字列にする。

例：

```text
k8Qf2xMzP1
```

公開URLはこうなる。

```text
https://example.com/e/k8Qf2xMzP1
```

---

## 7.2 event_slots

候補日時。

| カラム        | 型           | 説明        |
| ---------- | ----------- | --------- |
| id         | uuid        | 主キー       |
| event_id   | uuid        | events.id |
| starts_at  | timestamptz | 開始日時      |
| ends_at    | timestamptz | 終了日時      |
| label      | text        | 任意表示名     |
| sort_order | integer     | 表示順       |
| created_at | timestamptz | 作成日時      |

### 補足

候補日時は日付だけではなく、開始・終了時刻を持たせた方がよいです。

ただしUIでは、

```text
5/10 金 19:00〜
```

のように簡略表示してよいです。

---

## 7.3 respondents

回答者。

| カラム             | 型           | 説明          |
| --------------- | ----------- | ----------- |
| id              | uuid        | 主キー         |
| event_id        | uuid        | events.id   |
| display_name    | text        | 回答者名        |
| edit_token_hash | text        | 編集トークンのハッシュ |
| created_at      | timestamptz | 回答作成日時      |
| updated_at      | timestamptz | 回答更新日時      |
| deleted_at      | timestamptz | 論理削除        |

### 名前の重複

同じイベント内で同じ名前を完全禁止にすると、家族・同姓・同名で詰まる可能性があります。

推奨は、

* 同名がいたら警告する
* ただし登録は許可する

です。

例：

```text
同じ名前の回答があります。
自分の回答で間違いなければそのまま送信できます。
```

---

## 7.4 availabilities

各回答者の各候補日時への回答。

| カラム           | 型           | 説明                              |
| ------------- | ----------- | ------------------------------- |
| id            | uuid        | 主キー                             |
| respondent_id | uuid        | respondents.id                  |
| slot_id       | uuid        | event_slots.id                  |
| status        | text        | available / maybe / unavailable |
| created_at    | timestamptz | 作成日時                            |
| updated_at    | timestamptz | 更新日時                            |

### 制約

`respondent_id` と `slot_id` の組み合わせは一意にする。

```text
unique(respondent_id, slot_id)
```

これにより、1人が同じ候補に複数回答することを防げます。

# 8. 権限設計

## 8.1 管理者

管理者は自分が作成したイベントだけ操作できる。

できること：

* 自分のイベントを作成
* 自分のイベントを閲覧
* 自分のイベントを編集
* 自分のイベントを削除
* 自分のイベントの回答を閲覧
* 自分のイベントの回答を削除

---

## 8.2 回答者

回答者はログインしない。

できること：

* 公開URLを知っているイベントを見る
* 募集中のイベントに回答する
* 自分の編集トークンを持っている回答だけ編集する

できないこと：

* 他人の回答を編集
* 管理画面を見る
* 締切済みイベントに新規回答
* 削除済みイベントを見る

---

## 8.3 実装上の推奨

公開回答の作成・更新は、ブラウザから直接Supabaseに書き込むより、Next.jsのServer ActionまたはRoute Handler経由にするのが安全です。

理由は、編集トークン検証、締切チェック、イベント状態チェック、入力バリデーションをサーバー側で一元化できるからです。

Vercelでは環境変数をプロジェクト設定で管理できるため、Supabaseのサーバー用キーなどはクライアントに出さず、サーバー側だけで扱う設計にします。([Vercel][3])

# 10. バリデーション要件

## 10.1 イベント作成

| 項目   | ルール            |
| ---- | -------------- |
| タイトル | 必須、1〜80文字      |
| 詳細   | 任意、最大2000文字    |
| 候補日時 | 1件以上、最大50件     |
| 開始日時 | 現在以降を推奨        |
| 終了日時 | 開始日時より後        |
| 締切日時 | 設定する場合は現在より後   |
| 候補重複 | 同一開始・終了時刻は重複警告 |

---

## 10.2 回答

| 項目     | ルール               |
| ------ | ----------------- |
| 名前     | 必須、1〜30文字         |
| 回答     | 最低1つは選択必須にするかは要検討 |
| 候補     | 存在する候補のみ回答可能      |
| 締切     | 締切後は新規回答不可        |
| イベント状態 | open のときだけ回答可能    |

### 回答必須ルールについて

「全部行けない」という回答もあり得ます。

そのため、以下のどちらかにします。

### 推奨A

全候補に対して、初期状態を「行けない」にする。
タップしたものだけ「行ける」にする。

この場合、未選択がなくなるので簡単です。

### 推奨B

初期状態は未回答。
1つ以上選ばないと送信できない。

ただし、回答者が少し迷いやすくなります。

個人的には **推奨A** がよいです。
「行ける日だけ押してください」という体験にできるからです。

# 11. UI/UX要件

## 11.1 全体方針

* スマホファースト
* 1画面で迷わせない
* ボタンは大きく
* 候補日時はカード形式
* 回答送信ボタンは画面下部に固定してもよい
* 入力項目を増やしすぎない
* 管理画面は多少情報量が多くてもよい

---

## 11.2 回答画面のUX

候補日時は、チェックボックスよりもカードタップの方がよいです。

例：

```text
○ 5/10 金
  19:00〜21:00

× 5/11 土
  18:00〜20:00
```

タップすると、

```text
行けない → 行ける
```

に切り替わる。

将来 `maybe` を入れるなら、

```text
× → ○ → △ → ×
```

の順で切り替えられる。

ただしMVPでは `× → ○ → ×` で十分。

---

## 11.3 回答中の補助表示

回答者には以下を見せるとよいです。

```text
3件中 2件選択中
```

または、

```text
行ける日を2件選択しています
```

こうすると、選択できている感覚が明確になります。

---

## 11.4 送信ボタン

画面下部に固定表示するのがよいです。

```text
[ 2件選択して回答する ]
```

選択数が反映されると、安心感があります。

---

## 11.5 回答済み状態

同じ端末で再度開いた場合は、最初から自分の回答を復元する。

```text
前回の回答を表示しています。
変更したら「回答を更新する」を押してください。
```

ボタン文言も変える。

新規：

```text
回答する
```

更新：

```text
回答を更新する
```

# 12. 管理者側UX

## 12.1 イベント作成後

イベント作成後は、すぐに招待URLを見せる。

```text
イベントを作成しました。

招待URL
https://example.com/e/k8Qf2xMzP1

[ URLをコピー ]
[ 回答画面を開く ]
[ 管理画面へ ]
```

---

## 12.2 回答状況

管理者が一番見たいのは、最適な日程です。

そのため、詳細画面の上部にランキングを出す。

```text
一番集まりやすい候補

5/10 金 19:00〜21:00
5人が参加可能

[ この候補をコピー ]
```

その下に詳細表を出す。

---

## 12.3 締切

管理者はイベントを締め切れる。

締切後、回答者画面では以下を表示。

```text
この予定調整は締め切られました。
```

すでに回答済みの人の編集を許すかどうかは設定で決められるとよいですが、MVPでは **締切後は新規回答・編集とも不可** でよいです。

---

# 13. ステータス設計

## event.status

| ステータス    | 意味    |
| -------- | ----- |
| draft    | 下書き   |
| open     | 募集中   |
| closed   | 締切済み  |
| archived | アーカイブ |

MVPでは `draft` を省略して、作成したら即 `open` でもよいです。

最小構成はこれ。

```text
open
closed
archived
```

削除された候補の回答は無効化する。

# 15. セキュリティ要件

## 15.1 公開URL

公開URLは知っている人だけがアクセスできる設計にする。

ただし、URLを知っていれば誰でも開けるので、機密情報は書かない前提。

---

## 15.2 public_slug

連番IDは禁止。

悪い例：

```text
/e/1
/e/2
/e/3
```

良い例：

```text
/e/k8Qf2xMzP1
/e/Vm29qLa7xT
```

---

## 15.3 編集トークン

編集トークンは推測困難なランダム文字列にする。

DBには生のトークンを保存しない。
保存するのはハッシュのみ。

---

## 15.4 管理者認証

管理者はSupabase Authでログインする。

パスワードはアプリ側で直接保存しない。
Supabase Auth側で管理する。

---

## 15.5 RLS

SupabaseのRLSは有効にする。

基本方針：

* 管理者は自分の `owner_user_id` のイベントだけ操作可能
* 回答者は直接DBを自由に操作できない
* 公開回答はサーバー経由で検証する
* service role key はブラウザに絶対出さない

---

# 19. 受け入れ条件

## 管理者ログイン

* 正しいメール・パスワードでログインできる
* 間違った情報ではログインできない
* ログインしていない状態で管理画面に行くとログイン画面へ飛ぶ

---

## イベント作成

* タイトルと候補日時を入力するとイベントを作成できる
* 候補日時が0件の場合は作成できない
* 作成後に招待URLが表示される
* 招待URLをコピーできる

---

## 回答

* 回答者はログインなしで回答できる
* 名前が空だと送信できない
* 行ける候補日時を選んで送信できる
* 送信後に完了画面が出る
* 同じ端末から再訪問すると回答を修正できる

---

## 管理者の回答確認

* 管理者は回答者一覧を見られる
* 候補日時ごとの参加可能人数を見られる
* 最も参加可能人数が多い候補が分かる
* 回答者ごとの○×が分かる

---

## 締切

* 管理者はイベントを締め切れる
* 締切後は新規回答できない
* 締切後の回答画面には締切済み表示が出る

---

# 22. 個人的な設計判断

このアプリは、最初から高機能にしない方がいいです。

一番大事なのは、

```text
URLを開く
名前を入れる
行ける日を押す
送信
```

の速さです。

なので、MVPでは以下の判断がよいです。

| 判断      | 推奨                    |
| ------- | --------------------- |
| 回答者ログイン | 不要                    |
| 管理者ログイン | 必要                    |
| 回答ステータス | 最初は○/×                |
| 編集方法    | localStorage + 編集トークン |
| DB      | Supabase必須            |
| カレンダー連携 | 後回し                   |
| UI      | スマホファースト              |
| 候補表示    | カード型                  |
| 管理結果表示  | ランキング + 表             |

結論として、**Vercel + Supabase + Next.js でかなり相性がいいアプリ**です。
複雑にしすぎなければ、MVPはかなり短期間で作れる構成です。
